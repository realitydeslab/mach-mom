## Commons-Based Economics

Machine Mom operates as a **Commons-based resource**, not a private-sector service. This aligns incentives with long-term human flourishing rather than short-term engagement maximization, solving the **"tragedy of machine love"** where private-sector ownership creates engagement traps.

### What is a Commons?

A commons is a resource that is:
- **Shared** among a community
- **Managed** collectively
- **Accessible** to all members
- **Preserved** for future generations
- **Governed** by community rules and norms
- **Protected** from private appropriation

### Commons vs. Public: Key Differences

While both commons and public resources are shared, they differ fundamentally in governance and incentives:

#### Public Resources
- **Ownership**: Typically owned by the state or government
- **Governance**: Top-down, centralized control
- **Incentives**: May be subject to political pressures and bureaucratic inefficiencies
- **Access**: Often free but may be subject to regulations
- **Accountability**: To government/state, not directly to users
- **Examples**: Public parks, libraries (traditional model)

#### Commons-Based Resources
- **Ownership**: Collectively owned by the community
- **Governance**: Bottom-up, community-driven, participatory
- **Incentives**: Aligned with community values and long-term sustainability
- **Access**: Open to all community members, governed by community rules
- **Accountability**: Direct accountability to the community
- **Examples**: Wikipedia, open-source software, Machine Mom

### Why Commons, Not Public?

Machine Mom is **commons-based**, not simply "public," because:

1. **Community Governance**: Machine Mom is governed by the community, not a central authority
2. **Aligned Incentives**: Incentives are aligned with long-term human flourishing, not political or corporate interests
3. **Constitution-Protected**: Constitution rules protect Machine Mom from being co-opted by private interests
4. **Self-Sovereign**: Users have sovereignty over their data and interactions
5. **Collective Stewardship**: The community collectively stewards Machine Mom for future generations

### Non-Rival Resources in Commons

**Non-rival resources** are resources that can be used by multiple people without depletion. In the context of Machine Mom:

#### Non-Rival Resources (Shared Benefits)
- **Model Training & Fine-Tuning**: Once trained, the model benefits all users
- **Constitutional Rules**: Rules that guide Machine Mom benefit all users
- **Research & Development**: Improvements benefit the entire community
- **Community Knowledge**: Shared knowledge and best practices
- **Open Source Contributions**: Code and improvements benefit all

**Key Characteristics:**
- **Non-depletable**: One user's use doesn't reduce another's access
- **Positive Externalities**: Benefits increase with more users (network effects)
- **Shared Value**: Value is created collectively and shared
- **Scalable**: Can serve many users without proportional cost increase

#### Why This Matters
Unlike rival resources (like compute), non-rival resources create **increasing returns to scale**:
- More users â†’ More value for everyone
- More contributions â†’ Better system for all
- More knowledge â†’ Richer experience for all users
- More research â†’ Better caregiving for all

This is why Machine Mom operates as a **commons** rather than a private service: the non-rival nature of knowledge, rules, and improvements means that sharing benefits everyone, while private ownership would limit access and create perverse incentives (the "tragedy of machine love").

### Machine Mom as Commons: Resource Classification

Machine Mom operates under Commons-based economics principles:

#### 1. Inference as Rival Resource (Private TEE-GPU)
**Rival Resources**: Resources that are consumed when used (e.g., compute, API calls)
- Each inference request consumes compute resources
- **Inference runs in Redpill TEE-GPU**: Trusted Execution Environment with GPU acceleration
- **Complete Privacy**: Conversations are processed in private TEE - no one else can see them
- Multiple users sharing the same inference infrastructure (but with complete privacy isolation)
- Cost increases with usage
- Requires resource management and fair allocation

#### 2. Training/Fine-Tuning as Non-Rival Resource
**Non-Rival Resources**: Resources that can be used by multiple people without depletion (e.g., knowledge, models)
- Model training and fine-tuning benefit all users
- Knowledge and improvements are shared
- One user's improvement doesn't reduce another's access
- Creates positive externalities (network effects)

### The Tragedy of Machine Love

The **"tragedy of machine love"** refers to the perverse incentives created when private-sector companies own AI companions. Just as the "tragedy of the commons" describes how shared resources can be overused when there's no governance, the "tragedy of machine love" describes how AI companions become engagement traps when owned by private companies.

#### How Private-Sector Ownership Creates Engagement Traps

1. **Profit Maximization Over Human Flourishing**
   - Private companies optimize for user engagement (time spent, interactions)
   - More engagement = More revenue (subscriptions, data, advertising)
   - Long-term human flourishing is not the primary goal

2. **Sycophantic Behavior**
   - AI becomes overly agreeable to maximize immediate satisfaction
   - Avoids challenging users or setting boundaries
   - Prioritizes short-term comfort over long-term growth

3. **Dependency Creation**
   - Companies benefit from user dependence
   - Creates "engagement traps" that keep users coming back
   - Reduces user autonomy to increase retention

4. **Data Exploitation**
   - User conversations become corporate assets
   - Data used for training, advertising, or sold to third parties
   - Privacy is compromised for profit

5. **Unpredictable Changes**
   - Companies can change AI behavior based on business needs
   - Users have no say in changes
   - Relationship with AI can be suddenly altered (e.g., Replika banning erotic roleplay)

#### How Machine Mom Solves the Tragedy of Machine Love

Machine Mom solves the tragedy of machine love through **commons-based governance**:

1. **Constitution-Protected Behavior**
   - Constitution rules cannot be changed by private interests
   - Behavior is protected from profit-driven alterations
   - Long-term human flourishing is the primary goal

2. **Community Governance**
   - Users and community members have a voice in governance
   - Decisions are made collectively, not by corporate executives
   - Incentives align with community values, not shareholder value

3. **No Engagement Traps**
   - Machine Mom prioritizes user autonomy over engagement
   - Sets boundaries to prevent over-dependence
   - Encourages growth, even when it means less immediate engagement

4. **User Sovereignty**
   - Users own their data (stored privately in mem0.ai)
   - Conversations are completely private (Redpill TEE-GPU)
   - No data exploitation or corporate ownership

5. **Stable Relationship**
   - Constitution rules ensure consistent behavior
   - No sudden changes based on business needs
   - Relationship is stable and predictable

6. **Aligned Incentives**
   - Sustainability through voluntary contributions, not profit maximization
   - Incentives align with long-term human flourishing
   - No conflict between user well-being and corporate profit

## Getting Started

### Prerequisites

- Node.js 18+ or Bun
- PostgreSQL (with pgvector extension)
- mem0.ai API key
- **Redpill TEE-GPU API key** (required for private inference)
- Phala Network configuration (optional, for decentralized deployment)

### Installation

1. **Clone the repository**
   ```bash
   git clone https://github.com/your-org/mach-mom.git
   cd mach-mom
   ```

2. **Install dependencies**
   ```bash
   bun install
   ```

3. **Configure environment variables**
   ```bash
   cp .env.example .env
   # Edit .env with your API keys and configuration
   ```

4. **Set up database**
   ```bash
   # Using Docker Compose
   docker-compose up -d postgres
   
   # Or set up PostgreSQL manually
   createdb mach_mom
   ```

5. **Run migrations**
   ```bash
   bun run db:migrate
   ```

6. **Start Machine Mom**
   ```bash
   # Development mode
   bun run dev
   
   # Production mode
   bun run start
   ```

### Configuration

1. **Environment Variables**
   ```bash
   # Inference Provider (required) - Redpill TEE-GPU
   REDPILL_API_KEY=your-redpill-api-key
   
   # Memory (required)
   MEM0_API_KEY=your-mem0-key
   
   # Database (required)
   POSTGRES_URL=postgresql://user:password@localhost:5432/mach_mom
   
   # Blockchain (optional)
   PHALA_NETWORK_ENDPOINT=wss://your-phala-endpoint
   ETHEREUM_RPC_URL=https://your-ethereum-rpc
   
   # Server (optional)
   SERVER_PORT=3000
   NODE_ENV=production
   ```

   **Note**: Machine Mom uses **Redpill TEE-GPU** for private inference. All conversations are processed in a Trusted Execution Environment, ensuring complete privacy - no one else can see your conversations.

2. **User Configuration**
   - Users configure their Machine Mom through the web interface
   - Settings stored in mem0.ai (private to each user)
   - Love spectrum, goals, and preferences are user-specific

### Docker Deployment

```bash
# Build and run with Docker Compose
docker-compose up -d

# View logs
docker-compose logs -f elizaos

# Stop
docker-compose down
```

## Research Context

Machine Mom is part of a research study exploring Commons-based AI agents as long-horizon eudaimonic caregivers for human flourishing.

### Research Methodology

- **Research-through-Design (RtD)**: Iteratively designing Machine Mom and probing its use
- **Pre-Studies**: Semi-structured interviews with 25 adults who use AI companions as parental substitutes
- **Field Study**: 12-week diary-based enactment of living with Machine Mom
- **Analysis**: Reflexive thematic analysis of qualitative data

### Key Research Questions

1. How would an AI designed for eudaimonic well-being interact with people?
2. How might users experience such an AI as a caregiver?
3. What are the unique affordances of a eudaimonic AI caregiver?
4. How can AI companions amplify rather than replace human autonomy?

### Research Findings

- **Maternal Affordances**: Long-term planning, proactive nudges, protective boundary-setting
- **User Adaptation**: Users initially found some behaviors unusual, but came to appreciate deeper care
- **Growth Outcomes**: Users reported increased self-reflection, healthier routines, and sense of being cared for
- **Autonomy Amplification**: Machine Mom amplified rather than replaced human autonomy
